# orionchen.me — i18n Unfreeze Decision

- **Date**: 2026-06-10
- **Author**: huccct (Orion Chen)
- **Status**: Approved (decision recorded; implementation in progress)
- **Supersedes**: section 7 i18n row and section 10 i18n open item in
  `2026-05-17-personal-studio-design.md`
- **Scope**: Lift the "i18n deferred" freeze and add an English locale to
  the studio site

---

## 1. Why now (trigger)

There is **no concrete English audience signal**. The original spec
(2026-05-17, section 7 / section 10) required "concrete English
audience demand" before re-evaluating, with an annual cadence floor.

The trigger this time is **author preference**, recorded as such. No
GitHub issue, no inbound English readers, no overseas job application
in motion. This deviates from the spec's gate; the deviation is
acknowledged here so future-me does not re-litigate it.

If the work proves to be a maintenance drain (English chrome rotting
out of sync with Chinese, English routes 404'ing in the wild), the
fallback is **not** to re-translate — it is to roll back to a single
locale (see section 6).

## 2. Routing shape (and the SEO red line)

`DESIGN.md:95` and the project `CLAUDE.md` both forbid changing the
`/blog/[slug]` and `/tags/[tag]` URL shapes — the v1 site's SEO
inventory rides on them. This rules out the Next 16 official
recommendation of wrapping every route in `app/[lang]/...` (which
would force `/zh/blog/...` and break every existing inbound link).

Decision:

- `zh` is the **default locale, no path prefix**: `/`, `/blog/foo`,
  `/works/bar`, `/tags/tutorial` — exactly today's shape, untouched.
- `en` is **path-prefixed**: `/en`, `/en/blog/foo`, `/en/works/bar`,
  `/en/tags/tutorial`.
- **No automatic Accept-Language redirect.** Visitors land on the URL
  they typed; a manual EN ↔ 中 toggle in the site header swaps locale
  while preserving the rest of the path. Rationale: URL stability
  beats convenience here, and Google handles bilingual sites better
  when each URL canonicalizes to one locale instead of redirecting on
  every request.

## 3. Library choice — no `next-intl`

The 2026-05-17 spec listed `next-intl` as a "deferred i18n option"
(section 11). It is not adopted.

Reasons:

1. The site has two locales, not twenty. The Next 16 native
   `dictionaries/{zh,en}.json` + `getDictionary(locale)` pattern
   (documented at
   `node_modules/next/dist/docs/01-app/02-guides/internationalization.md`)
   covers every requirement here without an extra dependency.
2. The hard work is the 41 hand-written Chinese posts + 4 works,
   none of which a library can make easier; the locale mechanics are
   the easy 5%.
3. Smaller surface area to maintain when the maintenance budget is
   2h/week (spec section 8.1).

If a future need (ICU pluralization, RTL, complex date formatting)
appears, swap in then.

## 4. Translation policy — chrome only, never bodies

Hand-translated by the author:

- Site chrome: nav labels, footer, section headers, error pages
- List pages: `/blog`, `/works`, `/tags`, `/guestbook` framing copy
- Metadata: `siteConfig.description`, page titles, OG fallback text
- JSON-LD breadcrumb labels

**Not translated** by this work:

- Post bodies (the 41 `.mdx` files in `src/content/posts/`). The
  project `CLAUDE.md` rule "Auto-translate posts" stays in force for
  this exact reason. English posts ship as separate hand-written
  `*.en.mdx` files (or, if simpler, as posts with `lang: en` in
  frontmatter), at the author's pace.
- Work bodies (the 4 `.mdx` files in `src/content/works/`).
- The `/about` page narrative. The Chinese version contains
  author-specific cadence ("还没到第五幕", "麻将馆的阿姨") that
  machine translation would flatten. The English `/en/about` ships
  as a placeholder paragraph plus a link back to the Chinese version
  until the author writes a deliberate English self-introduction.

## 5. Content schema

Both `postSchema` and `workSchema` gain a `lang` field:

```ts
lang: z.enum(['zh', 'en']).optional().default('zh')
```

`getPublishedPosts(locale)` and `getVisibleWorks(locale)` filter by
`lang`. Today's 41 posts and 4 works default to `zh` without any
frontmatter change.

When `/en/blog` has no English posts (likely the steady state for a
while), the list page renders an empty state with a link back to the
Chinese archive — better than an empty list with no explanation.

## 6. Rollback plan

If maintenance proves untenable (English chrome consistently rots,
or zero English traffic after 6 months), the rollback is:

1. Delete the `src/app/en/` route tree
2. Delete `src/i18n/`
3. Remove the `LanguageToggle` component
4. Remove `lang` from schemas (or leave it, since the default is `zh`
   and unused fields are cheap)
5. Restore the original section 7 / section 10 wording

The `lang` field on content is the only schema migration; the rest is
pure deletion. There is no DB, no runtime state, no third-party
service contract to unwind. Re-evaluation cadence: annually, same as
the original spec required for unfreezing.
